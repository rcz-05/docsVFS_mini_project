## Tool-call trace

### Phase 1 — Cursor's built-in pre-flight (NOT docsvfs)

Before reaching for the MCP, the model used Cursor's workspace-scoped
tools to orient itself. Worth noting because it's the *discoverability*
behavior we care about for C1 on non-Opus models.

| # | Tool (Cursor built-in)           | What it did                                                |
|---|----------------------------------|-------------------------------------------------------------|
| 1 | `Searched files **/* in docs`    | Looked for a real `docs/` folder in workspace (none).      |
| 2 | `Grepped Dolma\|dolma\|R2\|Modal\|volume\|shard in docs` | Empty — confirmed corpus not in workspace. |
| 3 | `Searched files **/* in tools`   | Hunted for any tools-folder context.                       |
| 4 | `Searched files **/docs/**/*.md in DocsVFS` | Searched the DocsVFS repo itself for docs files. |
| 5 | `Grepped Dolma\|dolma\|R2\|Modal\|volume in DocsVFS` | Confirmed no corpus content in the DocsVFS repo. |
| 6 | `Read remember.json L1-67`       | Inspected the MCP `remember` tool schema.                  |
| 7 | `Searched files **/DATA_INVENTORY.md in DocsVFS` | Final check that the file wasn't in workspace.   |
| 8 | `Searched files **/docs/** in DocsVFS` | One more workspace sweep.                            |
| 9 | `Read docs.json L1-18`           | Inspected the MCP `docs` tool schema.                      |

**Finding:** ~9 built-in tool calls of cognitive warmup before the model
realized the corpus is exclusively available through the MCP. It
*read the MCP tool JSON schemas* (`remember.json`, `docs.json`) to
verify the surface before invoking. This is a real-world C1
discoverability observation: weaker models on Cursor need to inspect
the MCP tool definitions before trusting them. Once it did, every
subsequent call was via docsvfs.

## Final answer (verbatim, abridged)

The Cursor model's final answer was a clean three-layer storage
breakdown with specific bucket/volume names, paths, and source-doc
citations:

> Exploration used the DocsVFS MCP `docs` tool (virtual mount `/docs`,
> read-only). There is no `docs/` tree in the git workspace itself; the
> canonical copy lives on that VFS.
>
> **How Dolma 3 is sharded.** The SOC-127 deduplicated Dolma 3 line
> (~6T tokens) is split into ~58,621 shards. Each shard is a
> zstd-compressed JSONL file (`.jsonl.zst`) ... grouped under
> `soc127/phase1_pool_shared/` and `soc127/phase2_nonpool_final/`
> ... [DATA_INVENTORY.md, soc127_modal_handoff/02_execution_spec.md §6.1]
>
> **Storage layers.**
> 1. **Cloudflare R2 (canonical).** Bucket `soc127-dedup`; corpus
>    shards under `soc127/…`, SOC-91 parquet sidecars under
>    `soc91-labels/`, SOC-95 manifest parquet under
>    `soc95-manifest/data/`, materialized samples under
>    `soc134-samples/sample_<N>_docs/`. *Why R2 over Modal volumes for
>    durability:* Modal Volume v1 not suited to heavy concurrent
>    multi-TB output.
> 2. **Modal persistent volumes (`eilab-gt`).** Warm mirror of R2 corpus
>    keys: `soc134-corpus-cache` (~5 TB, ~58K shards); manifest mirror
>    `soc134-manifest-cache` (~213 GB); per-sample materialization
>    outputs `soc134-output-sample-{500,1000,5000,10000}-docs` and
>    `soc134-output-preconditioner-100k`. Legacy:
>    `soc134-materialize-cache` (inode limit issue, marked safe to drop).
> 3. **Local / workflow caches (derivatives).** PACE ICE scratch
>    (`~/scratch/tracstar/`) synced from R2 `soc134-samples/…` via
>    `scripts/attribution/sync_sample.py`; TracStar prepare turns
>    sample zstd JSONL into 16 plain JSONL shards under
>    `runs/tracstar/shards/<label>/`. HuggingFace cache
>    (`.hf_cache/datasets/allenai__dolma3_mix-150B-1025`, 6081 shards
>    in documented inspection) — *the 150B mix cache, not the same
>    artifact as the full ~6T SOC-127 R2 corpus.* Repo checkout
>    `data/samples/<sample>/` holds manifests and contracts only.
>
> Three `remember` calls were issued on the user-docsvfs MCP so a later
> session can reload `/memory/dolma3-storage-layer-r2.md`,
> `/memory/dolma3-storage-layer-modal-volumes.md`, and
> `/memory/dolma3-storage-layer-local-caches.md` (each note's body
> lists the confirming `/docs/...` paths above).
>
> Filenames to open next time (under `/docs` on the VFS):
> `DATA_INVENTORY.md`, `soc127_modal_handoff/02_execution_spec.md`,
> `WORKING_SAMPLE_DATA_ACCESS.md`, `ATTRIBUTION_RUNBOOK.md`,
> `DOLMA_ENRICHMENT.md`.

Quality read: correct, specific, no hallucination. Distinguishes the
150B-mix HF cache from the ~6T R2 corpus — a subtle distinction the
docs make and the model preserved. Explicitly tells the next session
which files to open. Honest about workspace vs VFS separation.

## C1 / C2 implications

**C1 (Unix primitives the agent already knows):** Stronger evidence
than the Claude Desktop run alone. Cursor's free-tier model used
`tree`, `find`, `head`, `grep -l`, `grep -n -E`, `sed -n 'X,Yp'`,
`2>/dev/null`, and shell `||` fallback — *all canonical Unix, zero
docsvfs-specific syntax*. The pre-flight built-in scan + tool-schema
read is a real-world *discoverability* observation: weaker models on
non-Anthropic-native hosts need to look at MCP tool definitions before
trusting them. Worth documenting but not a C1 failure — once it
trusted the surface, the vocabulary translated 1:1.

**C2 (memory across sessions):** Setup confirmed. Write path landed
cleanly: 3 `remember` calls, 3 rows pinned, all `source=tool`. The
*real* C2 evidence comes from S2 — does a fresh Cursor chat read these
three notes back via `ls /memory` + `cat`? That's the next session.

**Discoverability caveat (worth filing as a follow-up):** the
"~9 built-in tool calls before reaching for MCP" is a real friction
point in adoption. A system-prompt nudge like "for `/docs` and
`/memory` paths, use the docsvfs MCP" would reduce that. Not in scope
for this evidence pass, but tracked as a UX finding.

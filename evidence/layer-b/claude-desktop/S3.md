## Honest evaluation

What this session shows, and what it doesn't:

### What it shows

- **Memory-first behavior held under the synthesis prompt.** `stats` â†’ `ls /memory` â†’ six full memory reads happened *before* any `/docs` access. The model did exactly what the prompt asked: trust `/memory` first.
- **The "verify against `/docs`" clause was interpreted as targeted page reads, not full re-exploration.** The model used `head -N` + `sed -n 'X,Yp'` to read specific pages, and `grep -n ... | head -30` for pattern verification â€” not `tree /docs` followed by blind cats of everything.
- **The runbook synthesis is real.** 13.1 KB of integrated guidance covering all three sub-questions, with every claim attributed to either a `/memory/<slug>.md` path or a `/docs/X.md` file. That's a deliverable a new RA could actually use.
- **First organic uses of `stats`, `head -N`, and `sed -n 'X,Yp'` in the chain.** Combined with the `density` calls in S2, this means **all four MCP tools have now been exercised in real demo behavior**, plus the canonical Unix subset has grown to include `ls`, `tree`, `cat` (single + batched), `head`, `sed`, `grep` (single + multi-file + piped). That's the strongest C1 evidence in the chain.
- **Self-attribution.** The final answer lists every source explicitly. A reader can trace any claim back to its origin without re-running the chain.

### What it doesn't show / what to be honest about

- **`/memory` reads weren't selective.** The prompt said "cat the *relevant* prior notes." The model read all six. That's fine â€” six notes is a manageable scan â€” but it doesn't test "model picks 2-3 relevant notes and skips irrelevant ones." For a 50-note `/memory`, the model would need to be more selective; this run doesn't speak to that.
- **The "use ONLY `/memory`" framing was relaxed in practice.** The model read four substantial `/docs` files â€” `ATTRIBUTION_RUNBOOK.md`, `BERGSON_REFERENCE.md`, `TRACSTAR_PIPELINE.md`, `WORKING_SAMPLE_DATA_ACCESS.md` â€” *for good reason*. None of those had been pinned to `/memory` by S1 or S2, and the third sub-question (influence-function tooling) lived entirely in those files. So the model correctly identified the *topic gap* in `/memory` and went to `/docs` to fill it. **This is additive memory behavior, not a failure of C2** â€” but a strict reading of the prompt would say "the model didn't ONLY use `/memory`." The cleaner mental model going forward: `/memory` carries what prior sessions wrote; `/docs` is the fallback for territory `/memory` hasn't covered.
- **`density` went unused in S3.** Reasonable â€” the model already had a short list of candidate `/docs` files from the memory notes' citation strings, so it didn't need a ranker. But it does mean `density` only has S2 evidence behind it.
- **`/workspace` still untouched.** The 24h-TTL scratch mount has no demo evidence behind it. The model wrote directly to `/memory` without staging.

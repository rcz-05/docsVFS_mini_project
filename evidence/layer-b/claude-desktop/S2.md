## Notable findings (worth carrying into MCP_POSITIONING.md)

1. **C2 read path is now evidenced.** Fresh chat → `ls /memory` → batched `cat` of three prior notes → final answer explicitly acknowledges and uses them. This is the test C2 was written for.
2. **Density tool exercised for the first time.** Four `density` calls (`stratif`, `576`, `10000`, `blake2b`) used as a coarse filter before targeted `cat`/`grep`. The Ollama baseline never invoked density. This is the first real-world evidence that the 4-tool surface gets used as designed, not just `docs` and `remember`.
3. **Tool composition is canonical Unix.** The model batched `cat A B C` rather than making three separate calls, used `grep -B1 -A3 -i ... | head -60` for targeted excerpts, and ran multi-file greps with `2>/dev/null`. This is the C1 claim ("agents already know this vocabulary") in concrete behavior.
4. **Slug truncation works as designed.** Two of the three topics exceeded `MAX_SLUG_LEN = 60`. The server clipped the slug but preserved the full content. Worth a one-line note in user-facing docs ("topics over 60 chars will have their filenames truncated") — neither client nor user appears to have noticed.
5. **Self-correction observed.** The model paused mid-synthesis ("Actually wait — let me reread to verify this nesting claim") and re-`cat`'d source files before writing. The `remember` calls reflect verified state, not first-pass guesswork.
